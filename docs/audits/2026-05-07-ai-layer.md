# Audit — AI layer

> **TL;DR.** `server/ai.ts` is a clean, well-named *collection of nine independent OpenAI helpers* — but there is no generation pipeline. Every function reimplements `client.chat.completions.create(...)` inline; the three `draftReply` callers each rebuild the same 6-line context block by hand. The send side has a shared funnel (`aiAutoSend`), the generate side does not.

**Scope.** `server/ai.ts` (990 lines), `server/config.ts` (32 lines), AI call sites in `server/index.ts` (~14 sites across ~700 lines)
**Lines audited.** ~1,750

## Part 1 — Architectural breakdown

### Client + key/model resolution

A small, well-considered layer at the top of `server/ai.ts`. Settings (app.db) wins over env (`.env`); a per-key cache invalidates the OpenAI client on settings change without restarting the service.

```ts
// server/ai.ts:42-53
export function getOpenAI(): OpenAI {
  const key = effectiveApiKey();
  if (!key) throw new Error('OpenAI API key not configured. ...');
  if (_client && _clientForKey === key) return _client;
  _client = new OpenAI({ apiKey: key });
  _clientForKey = key;
  return _client;
}
```

This is the *only* shared infrastructure on the generation side. Everything below it duplicates.

### Capability functions (nine of them)

Each capability is one exported async function with: a private `*_SYSTEM` constant, an `Input`/`Result` interface, and an inline `client.chat.completions.create(...)` call.

```ts
// server/ai.ts:85-107  (representative shape — repeated 9 times)
export async function classifyIncoming(text: string): Promise<ClassificationResult> {
  const client = getOpenAI();
  const resp = await client.chat.completions.create({
    model: effectiveModel(),
    response_format: { type: 'json_object' },
    messages: [
      { role: 'system', content: CLASSIFY_SYSTEM },
      { role: 'user', content: `Message: ${text}` },
    ],
    max_tokens: 200,
    temperature: 0.2,
  });
  const raw = resp.choices[0]?.message?.content ?? '{}';
  const parsed = JSON.parse(raw);  // throws on malformed
  // ... validate shape, return typed result
}
```

The nine: `classifyIncoming`, `generateVoiceProfile`, `summarizeThread`, `extractRadarSignals`, `distillRadarProfile`, `extractCalendarEvent`, `extractAwayNote`, `evaluateRuleAgainstMessage`, `draftReply`. They share *no abstraction* — model resolution and `getOpenAI()` is the only common code.

### Drafting (`draftReply`) — the most complex capability

The only function with real prompt assembly: `buildSystemPrompt()` composes from voice profile, contact profile, address book context, calendar context, contact notes, temperament guidance, and away-mode guardrail. All other capabilities pass a fixed system prompt.

```ts
// server/ai.ts:806-873
function buildSystemPrompt(
  voiceProfile, contactNotes, contactProfile,
  addressBookContext, userAvailability, temperament, awayMode,
): string {
  const parts: string[] = [DRAFT_SYSTEM];
  if (voiceProfile?.trim()) parts.push(`\nVOICE PROFILE — ...`);
  if (contactProfile?.trim()) parts.push(`\nWHO YOU'RE TALKING TO — ...`);
  // ...
}
```

Notably, this builder lives in `ai.ts`, but the *inputs* to it are gathered three separate times in `index.ts` (see Findings below).

### Call sites (consumer side)

All AI imports live in **one** file — `server/index.ts:147-163` — pulling 11 names (functions + types). No AI imports leak into `db/`, `watcher.ts`, `send.ts`, or `integrations/`. That's good.

The 14 call sites split into two categories:

| Category | Sites | Pattern |
|---|---|---|
| HTTP route handlers | `/api/ai/classify`, `/api/ai/draft`, `/api/ai/summarize`, `/api/ai/voice-profile/regenerate`, `/api/radar/.../regenerate` | `requireAI(req, res)` gate → build context → call AI fn → optionally `createDraft(...)` → respond |
| Watcher-driven background | `triageInboundMessage`, `handleAwayModeMessage` (continuation), `handleSummonModeMessage`, `evaluateMessageAgainstRules`, `evaluateMessageForCalendar`, `evaluateMessageForRadar` | `if (!isAIConfigured()) return` gate → build context → call AI fn → persist + `sseBroadcast(...)` |

Both shapes are sane individually. The duplication is *across* sites within the same shape — see Findings.

### The send-side pipeline (for contrast)

Worth calling out because it's the model the generate side should follow:

```ts
// server/index.ts:1736-1762
async function aiAutoSend(opts: {
  handle: string; body: string;
  delayProfile: 'away' | 'summon' | 'off';
  abortIf?: () => boolean; logTag?: string;
}): Promise<{ sent: boolean; aborted: boolean; prefixedBody: string }> {
  const prefixedBody = withGaltPrefix(opts.body);
  if (opts.delayProfile !== 'off') { /* humanizing delay */ }
  if (opts.abortIf?.()) return { sent: false, aborted: true, prefixedBody };
  trackOurAutoSend(opts.handle, prefixedBody);
  await sendMessageViaAppleScript(opts.handle, prefixedBody);
  return { sent: true, aborted: false, prefixedBody };
}
```

Single funnel for every AI-generated send. Galt-prefix application, delay, abort-on-state-change, echo tracking, AppleScript invocation — all in one place. **This is the shape `generate()` should mirror on the inbound side.**

## Part 2 — Honest assessment

### What's working

- **One file owns AI; one file consumes it.** Imports go `server/ai.ts → server/index.ts` and nowhere else. No leakage into `db/`, `watcher.ts`, `send.ts`, integrations. Boundary is intact.
- **Key/model resolution is centralized and correct.** Settings-wins-over-env, per-key client cache, exposed source for `/api/health`. `apiKeySource()` is a small but real piece of operability — `server/ai.ts:27-31`.
- **Gating is consistent.** Every route checks `requireAI(req, res)`; every background path checks `if (!isAIConfigured()) return`. No path silently calls OpenAI without a key check.
- **`buildThreadFromMessages` and `isMeaningfulMessageText` are reused** correctly across `draftReply`, `summarizeThread`, away, and summon paths. Decoder fallbacks happen in one place — `server/ai.ts:947-990`.
- **Send-side pipeline is well-factored.** `aiAutoSend` (`server/index.ts:1736`) is a clean shared funnel — proves the team knows how to do this when they decide to. The AI-prefix helper `withGaltPrefix` is idempotent and centralized.
- **Prompt quality is unusually high.** The away-mode guardrail (`server/ai.ts:851-871`), radar extraction guidance, and per-temperament prose are specific, evidence-based, and conservative. Whoever wrote these took it seriously.
- **`distillRadarProfile` and `generateVoiceProfile` correctly preserve prior state** rather than regenerating cold — `EXISTING PROFILE (refine, don't replace cold...)`. That pattern should propagate to any future memory-bank work.

### Tradeoffs worth naming

The current design optimizes for **easy "add a new capability"** — drop a new function in `ai.ts`, import it in `index.ts`, ship. That's how nine capabilities accumulated in 990 lines without anyone losing the plot. The cost is paid every time you want to *change something across all capabilities*: retries, logging, cost tracking, prompt versioning, schema validation, model fallback, a brain/Firebase publish hook. Right now every cross-cutting concern is a nine-site sweep.

The send side already crossed this line — `aiAutoSend` exists because four call sites needed delay + abort + echo + AppleScript in lockstep, and the team correctly factored. The generate side hasn't crossed it yet because *most* cross-cutting concerns are still hypothetical. The Firebase substrate brainstorm in this branch makes them concrete: realtime publish on every AI call, prompt trace-back from the phone, cost dashboards across systems — all of those want one funnel.

Schema-validated outputs (Zod + OpenAI's structured outputs API) would also slot in naturally during a funnel refactor — and replace the seven hand-rolled `JSON.parse + typeof checks` blocks. That alone cuts ~80 lines of validation code.

## Bottom line

**Refactor before adding more.** The codebase has hit the point where the tenth capability function would inherit nine duplicated patterns — and the work the user is exploring (Firebase publish on every AI call, brain-context injection into drafts, cost tracking, prompt trace-back) makes the funnel mandatory, not optional. Do this *before* the brain wiring lands, not after.

Concrete shape that preserves what's working:

1. **`generate(opts)` funnel** in `ai.ts` — handles client, model, retry, timeout, abort, JSON parse, and (one) structured log line. Returns `{ raw, parsed?, usage, model, ms }`.
2. **Each capability function shrinks to ~10 lines** — builds prompt, calls `generate()`, validates result, returns typed result. No `client.chat.completions.create` outside the funnel.
3. **`resolveDraftInputs(handle, chatId, opts)` helper** in `index.ts` — collapses the three duplicated context blocks into one. Future memory/brain context plugs in here.
4. **`ai_calls` table** in `app.db` — one row per call. Powers cost tracking, "why did Galt do that" trace-back, and the future Firebase publish hook (publish events out of this table, not out of nine call sites).

The existing capability prompts are good and should not be touched in this refactor — the system prompts, away-mode guardrail, and per-temperament guidance are the highest-value content in `ai.ts`. The refactor is purely structural plumbing around them.

**Adjacent observations.** `server/index.ts` is at 2,550 lines and has accreted 14 AI call sites alongside the rest of the routes/watcher handlers. Splitting AI route handlers into `server/routes/ai.ts` and watcher-AI handlers into `server/watcher/handlers.ts` is overdue, but out of scope for this audit — file once via `/task` if it matters.
